# Full CMS i18n — All Hardcoded Strings to Sanity

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Move ALL hardcoded UI strings and page content into Sanity CMS so every text translates when switching locale.

**Architecture:** Create a `uiStrings` singleton document for shared UI chrome (nav, footer, forms, status labels, table headers, filters, CTAs). Create an `aboutPage` singleton for the About page content. Expand existing page schemas (`locationPage`, `residencesPage`, `masterplanPage`, `contactPage`, `homePage`) to cover all their hardcoded section content. Every component receives translated strings via props from its parent page — no hardcoded user-facing text remains.

**Tech Stack:** Sanity CMS, Next.js 15 App Router, TypeScript, `localeString`/`localeText` Sanity types, `getLocalizedValue()` helper.

**Key convention:** All localized fields use existing `localeString` (short) or `localeText` (long) types — objects with `{ en, he, ru, el }`. Components receive already-resolved strings (parent calls `getLocalizedValue(field, locale)`).

import { sanityClient } from "@/lib/sanity/client";
import { uiStringsQuery } from "@/lib/sanity/queries";
import type { UiStrings } from "@/lib/sanity/types";

let cached: UiStrings | null = null;

export async function getUiStrings(): Promise<UiStrings | null> {
  if (cached) return cached;
  try {
    cached = await sanityClient.fetch<UiStrings>(uiStringsQuery);
    return cached;
  } catch {
    return null;
  }
}
